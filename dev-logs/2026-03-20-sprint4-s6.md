# Dev Log: Sprint 4, Session 6 — DEF-021 Truncation Fix + Re-seed

**Date:** 2026-03-20
**Branch:** sprint-4

## What happened

Investigated DEF-021 (10,243-char truncation bug from Sprint 3.5). Traced the
full pipeline: segmentation text extraction, embed_text, insert_event, Supabase
REST API, postgrest-py, httpx. No truncation logic found anywhere. The
`max_tokens=4096` on the segmentation Claude call was the most actionable
code-level issue — increased to 16384 and added a `stop_reason` guard that
raises `SegmentationError` if the response is truncated.

Also added shared-boundary tolerance: when Claude returns two adjacent segments
sharing a boundary line (`start_line == prev_end_line`), the second segment's
start is silently adjusted to `prev_end + 1`. This was needed to process the
March 18 transcript.

Cleared production data (57 events, 6 clusters, 6 myths) and re-seeded from
both source transcripts. Results: 29 events, 7 clusters (6 seed + 1 dynamic
"The Argot"). No truncated notes — longest note is 25,568 chars, 4 notes
exceed the old 10,243-char limit.

Updated integration tests in test_clustering.py to tolerate dynamic clusters
in the production DB (filter by `is_seed` instead of asserting exact count).
