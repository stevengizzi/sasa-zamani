# Sprint 4, Session 5: Telegram Pipeline Integration

**Date:** 2026-03-19
**Sprint:** 4 — Data Quality + Significance Filtering
**Session:** 5 — Wire raw_input storage, significance filtering, and dynamic clustering into Telegram pipeline

## What Changed

1. **Pipeline reorder** — `process_telegram_update()` now runs label+significance
   generation before embedding. This lets the significance gate fire before the
   OpenAI embed call, saving API cost for low-significance messages.

2. **Raw input storage** — Every Telegram message is stored in `raw_inputs` (via
   `insert_raw_input`) before the significance check, so all inputs are preserved
   regardless of whether they become events.

3. **Significance filtering** — Messages with significance < `significance_threshold`
   (default 0.3) are stored in raw_inputs but not promoted to events. The return
   dict includes `reason: "below_significance"` and the `raw_input_id`.

4. **Dynamic cluster creation** — Replaced `assign_cluster()` with
   `assign_or_create_cluster()`, which creates a new cluster when the best
   similarity is below `cluster_join_threshold`.

5. **Archetype naming** — Added `maybe_name_cluster()` call after event insert.
   Wrapped in try/except so naming failures don't block the pipeline.

6. **Return dict extended** — All return paths now include `raw_input_id`.
